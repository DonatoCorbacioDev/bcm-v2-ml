# Demo — `/agent/insights` on a real (synthetic) portfolio

This is a real, captured run against the local Docker stack (`bcm-mysql`,
`bcm-ml`, and a local Ollama instance) — not a hand-written example. It shows
the full pipeline: rule-based risk scoring → ML risk scoring → forecast →
natural-language report, on the same synthetic dataset seeded by
`scripts/seed_synthetic_data.py`.

## How to reproduce this

```bash
# from bcm-v2-docker, with Ollama running locally
docker compose up -d mysql ml
ollama pull llama3.2   # if not already pulled

curl "http://localhost:8000/agent/insights?months=3&org_id=1"
```

No `Authorization` header is needed here because `INTERNAL_API_KEY` is unset
in local dev (see [MODEL_CARD.md](../../MODEL_CARD.md) and `app/security.py`)
— in the deployed system this endpoint is only reachable from the backend
(see the "not yet wired into the backend" note in [README.md](../../README.md)).

## The contract: why it's HIGH risk, concretely

Out of 41 contracts in the demo organization, `/risk-scores?org_id=1` ranked
**contract #78 — "Sbarbaro-Gibilisco e figli" (SYN-01-0008)** #2 overall:

| Field | Value |
|---|---|
| Status | `EXPIRED` (end date 2026-04-29, ~2 months before this run) |
| Business area | Construction |
| Total contract value | €56,719.70 across 4 monthly records |
| Manager assigned | **none** (`manager_id` is `NULL`) |

```json
{
  "contractId": 78,
  "customerName": "Sbarbaro-Gibilisco e figli",
  "riskScore": 0.6717,
  "level": "HIGH",
  "anomalies": ["EXPIRED"],
  "mlScore": 0.925,
  "mlLevel": "HIGH"
}
```

**Where 0.6717 comes from** (`app/services/risk_scoring.py`):

```
expiry_score = 1.0                         # status == EXPIRED
z_score      ≈ 0.54                        # back-computed: not flagged UNUSUAL_VALUE (|z| <= 2)
risk_score   = 0.6 * 1.0 + 0.4 * min(0.54/3, 1.0)
             = 0.6 + 0.4 * 0.179
             = 0.6717
```

The expiry alone already contributes 0.6 of the 1.0 scale — this contract
would be HIGH risk (>= 0.65 threshold) almost regardless of its financial
profile, because it's overdue. The amount is unremarkable for this
organization (z ≈ 0.54, well under the |z| > 2 anomaly cutoff), which is
exactly why `anomalies` contains only `EXPIRED`, not `UNUSUAL_VALUE`.

**The ML score (0.925) agrees, more strongly.** The classifier sees the same
`days_until_expiry` (negative, since it's overdue) and `status_code`, plus
`financial_std` and `num_financial_records` the rule-based score ignores —
in this case pushing its confidence that this is HIGH higher than the rule
alone would justify. See [MODEL_CARD.md](../../MODEL_CARD.md#2-contract-risk-scoring-get-risk-scores)
for why this agreement should be read as "second opinion," not independent
validation — both scores are ultimately measuring the same expiry signal.

**What no score captures:** this contract has no manager assigned. Nobody
in the organization is the accountable owner for chasing a renewal or
closing it out — that's a real operational risk the current risk-scoring
formula doesn't encode at all, since it only reasons about dates and
amounts. This is the kind of gap the "recommended actions" business
workflow (see the main project's roadmap) would actually need to close.

**Honest take on quality:** this is `llama3.2` (a small, locally-runnable
model, chosen so `/agent/insights` needs no external API key or cost) — the
report is coherent and grounded in the real numbers it was given (it doesn't
invent figures), but it's repetitive across bullet points and has the
occasional rough phrasing ("espirazione" instead of "scadenza" in a couple
of spots). It's a first pass, not a polished deliverable. A larger model
via `OLLAMA_MODEL` would read better; the rule-based/statistical numbers
behind it do not change either way — see
[MODEL_CARD.md](../../MODEL_CARD.md#4-natural-language-report-get-agentinsights)
for what the model does and does not influence.
